# Дневник разработчика / Контекст проекта

**Проект:** Интеллектуальная медицинская система (Грант СтС-626591).  
**Цель:** Создание AI-модуля для структурирования жалоб пациентов и реализации клинической логики на основе Клинических рекомендаций Минздрава РФ.

## 📌 Глобальное видение (из заявки)
- Развертывание локальной инфраструктуры в Docker (УЗ-2, ФЗ-152).
- Векторная база данных Qdrant для хранения клинических рекомендаций.
- Использование RuBioROBERTа для NLP-анализа жалоб.
- Внедрение DeepSeek-R1 в качестве интеллектуального ядра для принятия врачебных решений.

## 🛠 Что уже реализовано (Текущий код)
На данный момент полностью написан **Data Pipeline (конвейер подготовки данных)** и **Продвинутый RAG-поиск (Retrieval-Augmented Generation)**.

### 1. Этап извлечения и очистки данных (`main.py`)
- **`parser.py`**: Парсинг PDF-файлов клинических рекомендаций с помощью библиотеки `docling`. Извлекает текст и вырезает картинки/схемы.
- **`vision.py`**: Модуль оцифровки изображений. Отправляет вырезанные картинки в `gemini-2.5-flash` со строгим промптом (перевод схем в логические списки со стрелками `->`, запрет на галлюцинации).
- **`refiner.py`**: Модуль очистки текста через `gemini-2.5-flash`. Устраняет ошибки OCR, преобразует Markdown-таблицы в линейные логические цепочки. Сохраняет строгую структуру.

### 2. Этап векторизации (`vectorizer.py`)
- Разбиение очищенного JSON-текста на чанки (с помощью `MarkdownHeaderTextSplitter` и `RecursiveCharacterTextSplitter`).
- Реализован умный перехлест (overlap) между страницами, чтобы не терять контекст на стыке.
- Использование мощной модели **`BAAI/bge-m3`** для создания эмбеддингов.
- Загрузка данных в векторную БД **Qdrant**.

### 3. Этап поиска (`retriever.py`)
- **Перевод запроса:** Жалоба пациента написана простым языком. Сначала она отправляется в **YandexGPT** (prompt: \"переведи в строгий медицинский поисковый запрос\").
- **Поиск:** Гибридный поиск по базе Qdrant с использованием эмбеддингов `BGE-M3`.
- **Reranking (Переранжирование):** Полученные результаты прогоняются через `CrossEncoder ('BAAI/bge-reranker-base')` для точной сортировки по релевантности. Логиты переводятся в понятные проценты (0-100%).

## 🚀 Стек технологий в коде
- **Язык:** Python
- **LLM/VLM:** Google Gemini 2.5 Flash, YandexGPT (планируется DeepSeek-R1).
- **Embeddings & Reranker:** BAAI/bge-m3, BAAI/bge-reranker-base (HuggingFace/SentenceTransformers).
- **Векторная БД:** Qdrant.
- **Парсинг:** docling, fitz (PyMuPDF).

## 🔄 Журнал изменений (Changelog)

### [30.06.2026] Рефакторинг и актуализация процессов конвейера
В ходе анализа и развития кодовой базы были реализованы следующие новые процессы и изменения:

1. **Апгрейд VLM и изменение логики связей (`vision.py`):**
   - Оцифровка изображений переведена на более мощную модель **`google/gemini-2.5-pro`** через API OpenRouter для безошибочного распознавания сложных медицинских алгоритмов.
   - **Категорически запрещено** использование символов стрелок (`->`, `-->`, `=>`) в промптах. Все логические связи, переходы и ветвления блок-схем теперь описываются исключительно естественным языком, что радикально улучшает качество последующего векторного поиска.

2. **Изменение архитектуры перевода запросов (`retriever.py`):**
   - Процесс трансформации разговорных жалоб пациентов в строгие медицинские термины (`rewrite_patient_query`) переведен с YandexGPT на модель **`google/gemini-2.5-flash`** через OpenRouter. Это позволило добиться более жесткого удержания контекста симптомов.

3. **Повышение отказоустойчивости пайплайна (`refiner.py`):**
   - Интегрирован новый процесс **надежной очистки от Маркдауна**. Перед передачей текста в `json.loads()` код принудительно вырезает артефакты в виде сырых тегов ` ```json ` и ` ``` `. Это полностью устранило падения конвейера из-за случайного форматирования со стороны LLM.

4. **Оптимизация извлечения визуальных данных (`parser.py`):**
   - В настройки `PdfPipelineOptions` добавлены процессы `generate_table_images = True` (автоматическое вырезание таблиц как отдельных картинок для последующего анализа через VLM) и HD-масштабирование элементов документа `images_scale = 3.0` для повышения точности OCR.
